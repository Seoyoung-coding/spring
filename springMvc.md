# Spring MVC란?
Spring MVC는 Model-View-Controller 패턴을 기반으로 하는 웹 애플리케이션 프레임워크입니다.

## 0. 개발 흐름
1. 사용자 요청 → Controller가 받음
2. Controller → Service 호출 (비즈니스 로직 실행)
3. Service → Repository 호출 (데이터 조회/저장)
4. Repository → Database 접근
5. Database → Repository → Service → Controller → View → 사용자 응답

Service와 Repository 구현하기 & 3계층 연결


## 1. service
Service 계층의 설계
Service 계층은 Spring MVC 아키텍처에서 비즈니스 로직을 처리하는 핵심 부분입니다. 이 계층은 컨트롤러(Controller)와 데이터 접근 계층(Data Access Layer) 사이에 위치하며, 비즈니스 로직의 실행을 담당합니다.
//유저와 관련된 기능을 모아놓은 곳

@Service (애노테이션): 이 클래스가 비즈니스 로직(핵심 기능)을 수행하는 **'서비스 레이어'**임을 스프링에게 알려줍니다. 스프링은 이 애노테이션을 보고 "아, 이 객체는 내가 생성해서 관리해야겠구나"라고 판단하여 **Bean(객체)**으로 등록합니다.

1. 비즈니스 로직 중심
Service 계층은 비즈니스 요구사항을 반영한 메서드를 제공
컨트롤러가 직접 데이터를 처리하거나 비즈니스 규칙을 구현하지 않도록 함

2. 재사용성
각 서비스 메서드는 재사용 가능하도록 일반적인 비즈니스 기능을 수행
예: 사용자 인증, 데이터 검증 등은 여러 컨트롤러에서 공통적으로 요구되는 기능

3. 분리된 관심사
Service 계층은 데이터베이스, 웹 프레임워크, UI와 같은 기타 계층의 구현 세부사항으로부터 독립
인터페이스를 사용하여 구현 세부사항을 추상화

4. 트랜잭션 관리
Service 계층은 일반적으로 트랜잭션 관리의 경계를 정의
Spring의 선언적 트랜잭션 관리(@Transactional)를 활용하여 데이터 일관성 보장

5. 예외 처리
발생 가능한 예외를 적절히 처리하거나 변환하여 상위 계층으로 전파

1) 스프링이라는 프로그램은 코드에 붙어 있는 표시를 보고 필요한 기능을 자동으로 만들어 둔다. 이렇게 실제로 만들어진 결과물을 빈이라고 부른다.

이 빈을 등록하는 방법에는 두 가지가 있다.

첫 번째는 개발자가 설정 파일에서 어떤 기능을 어떻게 만들지 직접 지정해 주는 방법이다.
두 번째는 각 기능의 역할을 표시해 두면, 스프링이 알아서 해당 기능들을 찾아서 자동으로 만들어 주는 방법이다.

이 자동 방식에서는 서비스 역할이나 데이터 처리 역할을 가진 코드에 표시만 해 두면, 스프링이 필요한 기능들을 미리 준비하고 서로 연결까지 해 준다.

이렇게 스프링이 관리하는 기능으로 등록되면, 필요한 곳에서 “이 기능이 필요하다”고 표시만 해도 스프링이 자동으로 가져다 사용하게 해 준다.

=> @Service, @Repository, @Controller 추가해준다
